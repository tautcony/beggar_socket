## Context

目前连接相关动作在组件事件处理、use case 与 gateway 调用之间分散，常见问题包括：
- busy 状态与连接状态不同步；
- 失败路径后残留无效会话上下文；
- 不同入口触发连接时语义不一致。

连接编排需要成为应用层的一等能力，以支撑后续 Burner 流程解耦。

## Goals / Non-Goals

**Goals:**
- 用单一 connection use case 编排完整生命周期（list/select/connect/init/disconnect）。
- 建立可推导、可断言的连接状态机，统一状态迁移规则。
- 标准化连接流程错误与恢复策略，确保失败后可继续操作。

**Non-Goals:**
- 不在本变更中改变底层串口协议实现。
- 不重做 UI 视觉层，只调整调用入口与状态消费方式。
- 不覆盖非 Burner 业务域的连接场景。

## Decisions

### Decision 1: 显式连接状态机
- 定义连接状态枚举与合法迁移表，由 use case 统一驱动。
- UI 仅订阅状态，不直接维护连接阶段变量。
- 备选方案：沿用散落布尔值（`deviceReady`、`busy` 等）组合判断。
  - 放弃原因：状态组合不完备，难以覆盖边界场景。

### Decision 2: 连接命令按步骤编排但统一入口
- 对外提供统一连接入口（如 `prepareConnection` / `connectSelectedDevice`），内部拆分 list/select/connect/init 步骤。
- 每步返回标准结果，允许在失败时定位具体阶段。
- 备选方案：暴露大量低级方法让组件自行串联。
  - 放弃原因：编排逻辑回流到 UI，重复且不可控。

### Decision 3: 错误归一化与恢复策略内聚
- 连接相关错误在 use case 层统一映射为领域错误码。
- 失败后执行一致的会话清理策略（清 busy、失效旧 transport、保留可诊断日志）。
- 备选方案：各调用点独立处理错误。
  - 放弃原因：恢复策略不一致导致“假连接态”问题。

### Decision 4: 集成测试覆盖状态迁移
- 以场景测试验证关键迁移：成功连接、初始化失败、用户取消、断开后重连。
- 断言迁移后会话字段与 UI 可消费输出一致。
- 备选方案：仅写单元测试。
  - 放弃原因：跨层编排行为无法充分覆盖。

## Risks / Trade-offs

- [Risk] 状态机引入后初期改动范围较大，可能触发现有组件假设失效。  
  → Mitigation: 保留兼容映射字段，在一轮版本中双轨输出状态。

- [Risk] 连接失败归一化后，部分底层错误细节可能被压缩。  
  → Mitigation: 日志层保留原始错误上下文，用户层展示标准化消息。

- [Risk] 统一入口可能降低灵活性（调试时想单步执行）。  
  → Mitigation: 内部保留可测试的分步骤函数，仅不直接暴露给 UI。

## Migration Plan

1. 定义连接状态机与标准结果模型。
2. 实现 connection use case，并接入现有 gateway。
3. 迁移 `CartBurner` 中连接触发路径到新 use case。
4. 补齐连接生命周期集成测试并对齐断言。
5. 清理旧连接辅助逻辑与重复状态字段。

## Open Questions

- 连接状态是否需要持久化到 store，以支持页面切换后恢复？
- 对“自动重连”策略是否应在当前阶段支持，还是明确留到后续 change？
