## Context

当前 Burner ROM 写入在 `gba-adapter.ts` 与 `mbc5-adapter.ts` 中采用相同的高层流程：

1. 先根据目标范围计算涉及扇区
2. 若起始区域非空，先整体擦除这些扇区
3. 再按 page/chunk 线性写入

这个模型的问题是：一旦 `rom_program()` 或对应 transport 在某个 chunk 上超时，当前扇区可能已经被部分编程。此时简单“重试同一个 chunk”并不安全，因为 NOR Flash 通常无法在未重新擦除的情况下把已写 0 位恢复为 1；而“整个写入任务从头开始”又代价过高，并且 UI 无法区分普通写入与恢复阶段。

同时，读路径已经有 retry 与 channel resync 机制，说明项目已经接受“适配器内恢复策略 + 设置项可调”的模式。写入与扇区擦除恢复应沿用同样思路，但需要额外处理“部分成功导致扇区脏写”的设备语义。

## Goals / Non-Goals

**Goals:**
- 为 GBA/MBC5 ROM 写入提供可配置的 chunk 写入重试
- 为 GBA/MBC5 ROM 扇区擦除提供可配置的扇区擦除重试
- 在 chunk 写入失败且当前扇区可能已被部分编程时，先重擦除该扇区，再从该扇区起点恢复写入
- 保持 ROM 写入流程为“先完整擦除目标范围，再开始线性写入”，并把恢复成本约束在失败扇区内
- 为扇区级进度提供明确的擦除状态，让 UI 在擦除流程和恢复性重擦除时能区分待擦除、擦除中、已擦除，同时避免与普通写入进度语义混淆
- 为高级设置增加写入/擦除重试配置，并为新流程增加专项测试

**Non-Goals:**
- 不在本 change 中扩展到 RAM 写入恢复
- 不引入后台持久化 checkpoint 或断电续写
- 不改变协议层 packet 格式或 transport timeout 机制
- 不把 `ProgressInfo.state` 扩展为新的顶层生命周期状态；会话仍维持 `running/completed/error`

## Decisions

### D1: 以“扇区”为恢复边界，而不是按单个 chunk 直接重试

**选择**: 写入失败后，如果失败 chunk 所在扇区此前已成功写入过任意字节，则先重新擦除该扇区，再从该扇区起始地址重新写入整个扇区范围。

**理由**:
- NOR Flash 的写入失败意味着当前扇区可能已进入“部分编程”状态，直接续写同一 chunk 风险高。
- 以扇区为恢复边界与现有擦除单位一致，实现上也更容易复用 `calcSectorUsage()` 和 sector progress。
- 恢复代价被控制在单扇区，而不是整次写入范围。

**备选方案**:
- 直接重试失败 chunk：无法保证扇区仍可安全编程。
- 整个任务从头重来：恢复成本过高，用户体验差。

### D2: ROM 写入保持“先完整擦除，再线性写入”，恢复时只重做当前扇区

**选择**: 写入开始前先完整擦除本次写入目标范围；进入写入阶段后不再插入前推预擦除或写前采样读。只有当某个 chunk 写入失败时，才把当前扇区视为 dirty，执行恢复性重擦除并从该扇区起点恢复写入。

**理由**:
- 现有协议/固件擦除能力是阻塞式单通道，无法安全实现真正并行的“边擦边写”。
- 在当前约束下，预擦除窗口越大，越会打断写入吞吐并造成 UI 语义混乱。
- “先整段擦除，再线性写入”更符合当前设备能力，也更容易向用户解释。
- 失败恢复仍然保留在单扇区粒度，不必退回到整次任务从头开始。

**备选方案**:
- 前推预擦除 + 写前验空：在当前协议层下无法带来真实并行，反而会引入额外暂停。
- 完全按 chunk 现场擦除：调度更复杂，且同样无法解决单通道阻塞问题。

### D3: 擦除失败与写入失败都沿用 AdvancedSettings 可调重试模型

**选择**:
- 在 `AdvancedSettings.retry` 中新增 ROM write / ROM erase 的 count 与 delay 配置。
- 适配器在恢复逻辑中读取这些设置，并记录重试日志。

**理由**:
- 项目已有 `romReadRetryCount`, `romReadRetryDelayMs` 等模式，用户已有认知。
- 便于在不同设备/浏览器环境下调节恢复策略，不必硬编码。

**备选方案**:
- 只用固定常量：实现更快，但难以现场调参。
- 把重试只做在 adapter 内常量配置：不利于问题排查。

### D4: 扇区状态改为“擦除语义优先”的 phase-aware 枚举，而不是沿用泛化 `pending/processing/completed`

**选择**:
- 擦除维度至少显式区分 `pending_erase | erasing | erased`
- 恢复性重擦除在模型上归入擦除阶段，而不是伪装成普通 processing
- 普通写入流程继续沿用 `pending | processing | completed`，避免图例和当前扇区高亮在写入期间混入擦除语义
- 顶层 `ProgressInfo.type` 仍保持 `write` 或 `erase`
- 通过 `detail` 文案说明当前是在“完整预擦除 / 恢复性重擦除 / 重试写入”

**理由**:
- 你已经明确指出，从擦除角度看，扇区状态必须至少具备“待擦除、已擦除、擦除中”三态。
- 这比 `pending/processing/completed` 更不含糊，避免 UI 把“已擦除”和“已写入完成”混为一谈。
- 顶层 operation type 若新增 `recovering-write`/`re-erasing`，会放大 UI 改动面并影响会话流转。
- 现有进度弹窗和 session 只需要知道任务仍在运行，无需单独生命周期状态。

**备选方案**:
- 新增 `ProgressInfo.state = recovering`：会波及 `useCartBurnerSessionState`、modal 关闭规则和更多现有测试。
- 继续沿用 `pending/processing/completed`：无法满足你对擦除语义的要求，也会让恢复性擦除语义模糊。

### D5: 先在 ROM 写入路径落地，且同时覆盖 GBA 与 MBC5 适配器

**选择**: 恢复逻辑先实现于 `GBAAdapter.writeROM()` 与 `MBC5Adapter.writeROM()`，并分别复用各自的 `eraseSectors()` / bank switching / program 命令。

**理由**:
- 两个适配器目前结构高度相似，适合以同一算法分别落地。
- 用户看到的失败日志来自 ROM 写入路径，这里是最高优先级。
- 若只修一个适配器，会造成运行时行为不一致。

**备选方案**:
- 只改 MBC5：交付更快，但留下 GBA 不一致行为。
- 先抽象公共类再实现：设计更整洁，但当前目标是尽快恢复可写性。

### D6: 恢复前的“是否已部分写入”判断以会话内写入游标为准，不依赖回读探测

**选择**: 只要失败 chunk 所在扇区的起始地址小于当前已成功写入游标，就认为该扇区可能已被部分写入，需要重擦除。

**理由**:
- 会话内我们已经知道哪些 bytes 在本次任务里成功下发过。
- 失败后立即做额外回读探测会放大 transport 不稳定场景下的复杂度。
- “宁可多擦一次当前扇区，也不冒险直接续写”更安全。

**备选方案**:
- 通过回读判断是否真的脏写：更精细，但在 timeout 场景下常常同样不可靠。

### D7: 不在正常写入路径增加写前采样读

**选择**: 常规写入阶段不再增加写前采样读；写安全性依赖前置完整擦除，而不是在每个 chunk 前额外回读验证。

**理由**:
- 当前协议是单命令通道，写前采样读会直接打断写入吞吐。
- 设备侧读超时会把采样读本身变成新的失败源，收益不如成本明确。
- 在“先完整擦除”的流程下，恢复性逻辑只需要处理真正的写失败和擦除失败。

**备选方案**:
- 伪随机采样：增加额外读负担，且不能实现真正并行。
- 全块回读确认：开销更高，不适合当前路径。

## Risks / Trade-offs

- [恢复逻辑会拉长单扇区失败时长] → 通过可调 retry count/delay 控制上限，并把恢复细节写入日志
- [擦除态命名调整会影响 sector visualization 与国际化] → 将状态扩展集中在 `progress-info`、`ProgressReporter` 和 `SectorVisualization`，补齐 UI 测试/快照
- [完整预擦除会拉长写入前等待时间] → 用明确的擦除进度与可配置擦除重试来保证可观测性，并避免写入阶段再被擦除打断
- [GBA/MBC5 两套实现可能再次分叉] → 先统一设计与测试矩阵，后续如重复度过高再抽公共 helper
- [擦除重试可能掩盖真实硬件坏块] → 达到上限后仍返回失败，并在错误消息中保留“擦除重试耗尽”的明确原因

## Migration Plan

1. 先补 OpenSpec requirements，明确 ROM 写入恢复与进度状态契约。
2. 扩展 `AdvancedSettings` 与设置面板，增加 ROM write / ROM erase retry 配置。
3. 扩展 progress 类型、`ProgressReporter` 与 `SectorVisualization`，支持 `pending_erase`、`erasing`、`erased` 等擦除态命名，并与普通写入态区分展示。
4. 在 `MBC5Adapter.writeROM()` / `eraseSectors()` 中落地“先整段擦除，再写入；失败扇区局部恢复”算法与测试。
5. 将同一算法迁移到 `GBAAdapter.writeROM()` / `eraseSectors()` 并对齐测试。
6. 跑 adapter/progress/full web-client 测试与构建；硬件验证再确认真实超时恢复体验。
