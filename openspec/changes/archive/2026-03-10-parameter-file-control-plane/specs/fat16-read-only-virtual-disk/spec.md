## MODIFIED Requirements

### Requirement: Device exposes ROM and RAM subdirectories as fixed virtual directories
The system SHALL expose `ROM` and `RAM` as fixed subdirectories backed by firmware-defined directory entries rather than host-created filesystem state, and those subdirectories MAY contain additional fixed parameter-group subdirectories used by the control plane.

#### Scenario: Host opens ROM directory
- **WHEN** the host reads the `ROM` directory
- **THEN** the device returns the firmware-defined directory entries for the ROM view without requiring dynamic filesystem allocation

#### Scenario: Host opens RAM directory
- **WHEN** the host reads the `RAM` directory
- **THEN** the device returns the firmware-defined directory entries for the RAM view without requiring dynamic filesystem allocation

#### Scenario: Host opens a control-plane parameter directory
- **WHEN** the host reads a firmware-defined parameter subdirectory under `ROM` or `RAM`
- **THEN** the device returns the fixed directory entries for that parameter group without requiring dynamic filesystem allocation

### Requirement: Informational files are presented as virtual text files with explicit write semantics
The system SHALL present `INFO.TXT` and `STATUS.TXT` as virtual text files whose contents are generated by firmware at read time, and SHALL preserve their declared control-plane write semantics.

#### Scenario: Host reads INFO.TXT
- **WHEN** the host reads `INFO.TXT`
- **THEN** the device returns a readable text payload describing device identity or capability summary

#### Scenario: Host reads STATUS.TXT
- **WHEN** the host reads `STATUS.TXT`
- **THEN** the device returns a readable text payload describing current device status

#### Scenario: Host attempts to write INFO.TXT
- **WHEN** the host writes to `INFO.TXT`
- **THEN** the device rejects the write or preserves read-only semantics without mutating device state

#### Scenario: Host writes STATUS.TXT when it is not declared writable
- **WHEN** the host writes to `STATUS.TXT` and that file is defined as read-only by the active control-plane design
- **THEN** the device preserves read-only semantics without mutating device state

### Requirement: ROM directory exposes metadata and control-plane files
The system SHALL expose ROM metadata files and any firmware-defined ROM control-plane files through fixed entries in the `ROM` directory.

#### Scenario: Host lists ROM directory
- **WHEN** the host reads the `ROM` directory
- **THEN** the directory listing includes `CURRENT.GBA`, `CFI.TXT`, and any fixed ROM control-plane files or subdirectories defined by firmware

#### Scenario: Host reads CFI.TXT
- **WHEN** the host reads `/ROM/CFI.TXT`
- **THEN** the device returns a readable text payload derived from the cartridge CFI query data

#### Scenario: Host reads a ROM control-plane text file
- **WHEN** the host reads a firmware-defined ROM control-plane text file such as `SELECT.TXT` or `CONFIG.TXT`
- **THEN** the device returns a readable text payload describing the writable keys or current pending values for that ROM parameter surface

### Requirement: Control-plane writes are limited to firmware-defined writable files
The system SHALL limit host writes to the specific control-plane files declared writable by firmware rather than allowing general FAT file mutation.

#### Scenario: Host writes a declared writable control file
- **WHEN** the host overwrites a firmware-defined writable control file such as `SELECT.TXT` or `CONFIG.TXT` with valid content
- **THEN** the device routes the write through the corresponding control-plane parser instead of mutating FAT allocation state or cartridge payload data

#### Scenario: Host writes any non-writable virtual file
- **WHEN** the host writes `INFO.TXT`, candidate option files, `CURRENT.GBA`, `CURRENT.SAV`, or any other fixed view that is not declared writable
- **THEN** the device preserves existing virtual-file semantics and does not mutate cartridge payload data
