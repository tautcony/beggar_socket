## Context

当前设备已具备通过 `USB CDC` 私有协议访问卡带 ROM/RAM 的能力，底层总线访问由 `cart_adapter.c` 提供，协议解析与操作流程主要集中在 `uart.c`。本 change 的目标不是引入完整可写文件系统，而是先建立一个**固定布局、只读、可枚举的虚拟 FAT16 磁盘骨架**，用最小风险验证以下关键前提：

- `STM32F103` 在当前资源约束下能否稳定暴露 `USB MSC` 设备
- 固定 `FAT16` 布局是否能被主流桌面系统稳定识别
- 目录、簇、LBA 和虚拟文件视图之间的查表模型是否足够简单、可控
- `/ROM/CURRENT.GBA` 与 `/RAM/CURRENT.SAV` 是否可以作为只读文件窗口映射到底层卡带读取能力

当前约束包括：

- MCU RAM 和 Flash 较紧张，不能引入通用 FAT 文件系统库或复杂缓存层
- 当前卡带访问逻辑强依赖时序，不应在第一阶段同时叠加写路径和复杂任务状态机
- 操作系统对 MSC 设备的访问是扇区级随机读写，因此必须让 FAT 布局与文件视图映射尽可能静态、可预测
- 后续阶段会扩展参数文件与写窗口，因此本阶段的目录结构与簇布局要为未来预留稳定路径

利益相关方包括：

- 固件开发者：需要一个稳定、易调试的 MSC/FAT16 骨架
- 上位机或工具开发者：希望将来的交互统一到文件系统语义
- 终端用户：希望最终设备可被识别为“插入即见”的可移动磁盘

## Goals / Non-Goals

**Goals:**
- 提供一个可被系统识别的固定布局 `FAT16` 只读虚拟磁盘
- 建立最小目录结构：根目录、`ROM/`、`RAM/`
- 提供只读信息和状态文件，如 `INFO.TXT`、`STATUS.TXT`
- 提供只读数据窗口文件 `/ROM/CURRENT.GBA` 与 `/RAM/CURRENT.SAV`
- 建立 `LBA -> cluster -> file view` 的静态查表机制
- 为后续参数文件与上传文件预留目录和 cluster 布局扩展空间
- 尽量复用现有 ROM/RAM 读取能力，避免在本阶段引入新的设备访问语义

**Non-Goals:**
- 不实现任意文件创建、删除、重命名或动态 FAT 分配
- 不实现任何写卡能力或提交工作流
- 不在本阶段引入参数写入口、`SELECT.TXT` 或 `CONFIG.TXT`
- 不在本阶段实现 `ROM/UPLOAD.GBA` 或 `RAM/UPLOAD.SAV`
- 不要求第一阶段就保留现有 CDC 与 MSC 的复合设备形态；是否双类并存可留待后续权衡
- 不处理持久配置保存问题

## Decisions

### 1. 采用固定布局 FAT16，而不是通用 FAT 实现

**Decision:** 使用编译期定义的固定 `FAT16` 布局，Boot Sector、FAT 表、根目录、子目录和文件簇链全部静态生成。

**Rationale:**
- 当前目标只是提供可预测、可枚举的虚拟磁盘骨架，不需要完整文件系统能力
- 固定布局可让实现退化为查表与视图生成问题，更适合 `STM32F103` 资源条件
- 可显著降低与主机系统扇区级访问语义之间的复杂交互风险

**Alternatives considered:**
- 使用通用 FAT 库：实现和调试成本过高，且为后续写支持引入过早复杂度
- 自定义非 FAT 文件系统：主机兼容性差，违背“可移动磁盘”目标

### 2. Cluster size 固定为 1 sector（512B）

**Decision:** 第一阶段使用 `512B/sector`、`1 sector/cluster` 的最小簇粒度。

**Rationale:**
- 计算简单：`lba = data_lba + (cluster - 2)`
- 小文本文件和状态文件天然适合 1 cluster 占用
- 方便把控制文件、状态文件、目录文件都映射到固定单扇区 handler
- 对虚拟大文件窗口来说，cluster 更小意味着 offset 转换更直接

**Alternatives considered:**
- 更大 cluster：能减少 FAT 项数量，但会增加 offset 计算复杂度，也不利于小文件视图映射

### 3. 目录结构在第一阶段即按最终形态预留

**Decision:** 即使第一阶段只实现只读能力，目录结构和部分目录项也应按最终方案的主骨架规划，例如保留 `ROM/`、`RAM/` 以及参数目录的扩展空间。

**Rationale:**
- 后续阶段依赖固定路径与稳定簇布局，越早稳定越能减少迁移成本
- 主机侧如果未来见到同一路径语义发生大变更，兼容性和调试都更困难
- 只读候选项文件本身也可作为后续参数面板的视觉预演

**Alternatives considered:**
- 第一阶段只暴露极简目录，后续再扩展：短期简单，但后续会引入目录项、簇号和路径语义变化

### 4. 大文件使用“文件窗口视图”，而非本地存储文件

**Decision:** `/ROM/CURRENT.GBA` 与 `/RAM/CURRENT.SAV` 在 FAT 中表现为普通只读文件，但实现上是**按 offset 动态映射到底层卡地址的只读窗口**。

**Rationale:**
- 当前 MCU 没有能力把整张 ROM/SAV 镜像保存为本地文件
- 现有底层已经具备按地址读取卡带内容的能力
- 只读窗口模型和未来的上传窗口模型在实现结构上可复用“文件视图”概念

**Alternatives considered:**
- 先整包 dump 到 MCU 本地缓冲再返回：内存不足，不现实
- 先导出到外部存储：当前硬件没有对应介质

### 5. 引入显式 `file view` 抽象，而不是把逻辑散落在 MSC handler 中

**Decision:** 在 MSC 层之下引入统一的 `file view` 抽象，每个逻辑文件都有独立 `read` 行为，MSC 只负责把 `LBA` 路由到对应文件视图。

**Rationale:**
- 让 `USB MSC` 层保持薄，职责清晰
- 方便后续为参数文件、控制文件、上传窗口复用同一抽象
- 减少把 FAT 解析、目录项、设备访问和文本生成搅在一起的风险

**Alternatives considered:**
- 在 `READ10` 里直接硬编码所有路径逻辑：短期看似快，但后续难维护，且不利于扩展到参数/控制文件

### 6. 抽取 `cart_service` 只读接口，而不是直接复用 `uart.c`

**Decision:** 第一阶段开始就把 `ROM/RAM` 的读取能力抽成面向文件窗口的服务接口，例如：
- `cart_service_read_rom(offset, buf, len)`
- `cart_service_read_save(offset, buf, len)`
- `cart_service_get_rom_size()`
- `cart_service_get_save_size()`

**Rationale:**
- `uart.c` 当前是协议解析和设备操作耦合结构，不适合直接成为 MSC 文件视图实现层
- 服务层抽象可同时服务 CDC 和未来 MSC 路径，减少逻辑复制
- 后续阶段的写窗口和状态机也会依赖相同服务边界

**Alternatives considered:**
- 直接在 MSC 代码里调用 `cart_adapter.c`：会把设备总线细节泄露到文件系统层
- 复用 `uart.c` 命令处理：协议耦合过重，不符合职责分层

### 7. 第一阶段优先选择“MSC-only 骨架”，CDC 共存延后决策

**Decision:** 设计上先把只读虚拟盘视为一个可独立实现的 `MSC-only` 骨架；是否与现有 CDC 组成复合设备，留到后续阶段在资源与调试需求更明确时再决定。

**Rationale:**
- 只读磁盘的主要目的是验证 MSC/FAT16 骨架，不需要一开始就叠加 USB 复合设备复杂度
- 当前资源紧张，先缩小变量范围更有利于快速验证
- 后续若需要调试或兼容旧协议，再在设计上增配 CDC

**Alternatives considered:**
- 第一阶段就做 `MSC + CDC`：功能更完整，但会把问题空间扩大，增加 USB 描述符与内存占用复杂度

## Risks / Trade-offs

- **[主机兼容性差异]** → 不同系统对 FAT16/MSC 的目录访问、预读取和缓存行为可能不同；通过固定布局、最小目录项集和只读行为降低复杂度，并优先在目标系统上做枚举验证。
- **[ROM 读取性能不足]** → `/ROM/CURRENT.GBA` 作为文件窗口可能受到底层总线速度影响；通过只读阶段先测量吞吐，必要时后续增加读缓存或限制导出体验预期。
- **[资源占用过高]** → MSC、FAT16 映射和现有固件共存可能逼近 `STM32F103C8` 的 Flash/RAM 上限；通过静态表、小 cluster 结构和薄视图层减小开销，并把写路径延后。
- **[目录结构过早冻结]** → 第一阶段提前预留目录骨架会减少后续灵活性；通过只冻结主路径和核心目录，允许后续在未暴露区域追加文件。
- **[底层读取抽象不足]** → 若 `cart_service` 抽取不干净，可能导致 CDC 和 MSC 双路径逻辑分叉；通过先围绕只读场景提炼最小服务接口，降低抽象失败风险。

## Migration Plan

1. 先在设计上确定 FAT16 最小参数：sector 大小、cluster 大小、FAT 大小、根目录项数量、数据区起始 LBA。
2. 定义固定目录树和 cluster 分配表，生成编译期常量布局。
3. 实现 `virtual_disk` 的最小只读路由：
   - Boot Sector
   - FAT1/FAT2
   - Root Directory
   - Data Region
4. 实现 `file view` 只读抽象，并接入：
   - `INFO.TXT`
   - `STATUS.TXT`
   - `/ROM/CURRENT.GBA`
   - `/RAM/CURRENT.SAV`
5. 从 `uart.c`/现有逻辑中抽取只读 `cart_service` 接口，避免文件视图直接依赖协议层。
6. 在目标主机系统上验证：
   - 设备是否稳定枚举为 MSC/FAT16
   - 根目录和子目录是否正常显示
   - 大文件只读导出是否可行
7. 若验证失败，可回滚到仅保留现有 CDC 固件，不影响现有烧录链路。

## Open Questions

- 第一阶段是否需要在目录中就暴露未来的参数子目录，还是只保留 `ROM/` 和 `RAM/` 两层骨架？
   - 只保留两层
- `CURRENT.GBA` 和 `CURRENT.SAV` 的文件大小应完全依赖运行时探测，还是先使用保守固定上限？
   - 先使用保守固定值，在编码中写好注说明后续改动点
- 若主机对只读大文件进行激进预读，当前卡带读取吞吐是否足以支撑可接受体验？
- 第一阶段验证完成后，是否优先引入 `MSC + CDC` 复合设备，还是继续保持 `MSC-only` 演进？
