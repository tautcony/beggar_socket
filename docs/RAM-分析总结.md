## 主要内存占用分析

### 1. UART 通信缓冲区（最大占用）
```
uart.c 中的全局变量：
- cmdBuf[5500]      : 5,500 字节
- responBuf[5500]   : 5,500 字节
────────────────────────────────
  小计              : 11,000 字节 (53.7%)
```

**说明**：
- 这是最大的单一内存消耗者
- 用于USB虚拟串口通信
- 支持最大5KB的协议数据包

### 2. 卡带服务RAM任务（流式架构）
```
cart_service.c 中的全局变量：
- g_cart_service_ram_job : 620 字节 (3.0%)
  └─ sector_buffer[512]  : 512 字节（FAT16扇区）
  └─ error_message[96]   : 96 字节
  └─ 其他状态字段        : 12 字节
```

**重大改进**（2026-03-10）：
- ❌ 旧版本：32KB 缓冲区（物理不可能！）
- ✅ 新版本：620 字节（98% 内存减少）

### 3. 系统预留内存
```
链接器脚本中定义：
- 最小堆（Heap）   : 512 字节 (2.5%)
- 最小栈（Stack）  : 1,024 字节 (5.0%)
────────────────────────────────
  小计             : 1,536 字节 (7.5%)
```

### 4. USB 设备库内部
```
估算：
- USB endpoint 缓冲区
- USB 控制传输缓冲区
- 设备描述符存储
────────────────────────────────
  估计             : ~400 字节 (2.0%)
```

### 5. 其他全局变量
```
估算：
- 外设句柄（UART, USB, GPIO等）
- 配置数据
- 状态变量
────────────────────────────────
  估计             : ~2,000 字节 (9.8%)
```

## 关键发现

### ✅ 积极方面

1. **流式架构成功实施**
   - 解决了32KB缓冲区不可能的问题
   - 内存占用从32KB减少到620字节（98%减少）
   - 文档：`docs/RAM-UPLOAD-WORKFLOW.md`

2. **代码可以正常运行**
   - 约24%的RAM仍然可用
   - 足够应对一般的栈增长和动态分配

### ⚠️ 需要注意

1. **UART缓冲区占用过大**
   - 11,000字节占用了53.7%的RAM
   - 如果协议实际不需要5KB包，可以考虑减小

2. **预留内存较小**
   - 堆+栈仅1.5KB
   - 实际运行时栈可能超过最小值
   - 需要避免在函数中声明大型局部数组

3. **需要精确测量**
   - 当前是静态分析估算
   - 建议生成 `.map` 文件获得精确数据
   - 建议实现运行时内存监控

## 优化建议

### 高优先级

**1. 考虑减少UART缓冲区大小**
```c
// 当前
uint8_t cmdBuf[5500];        // 5,500 字节
uint8_t responBuf[5500];     // 5,500 字节

// 建议：如果协议实际只需4KB
uint8_t cmdBuf[4096];        // 4,096 字节
uint8_t responBuf[4096];     // 4,096 字节
// 节省：2,808 字节 (13.7%)
```

**2. 实现缓冲区共享**
```c
// cmdBuf 和 responBuf 不同时使用
// 考虑使用单个缓冲区
uint8_t uartBuffer[5500];    // 共享缓冲区
// 节省：5,500 字节 (26.8%)
```

### 中优先级

3. 优化栈使用（避免大型局部数组）
4. 使用DMA减少内存拷贝
5. 代码优化（`-Os` 编译选项）

## 如何获取精确数据

### 方法1：编译生成 .map 文件

```bash
# 在CMakeLists.txt或Makefile中添加
-Wl,-Map=output.map

# 然后查看
arm-none-eabi-size -A your_firmware.elf
```

**map文件会显示**：
- 每个段的确切大小
- 每个全局变量的地址和大小
- 堆栈的确切位置

### 方法2：运行时监控

在代码中添加：
```c
extern char _end;        // BSS段结束（堆开始）
extern char _estack;     // 栈顶（RAM结束）

void print_memory_usage(void)
{
    // 计算已使用的RAM
    uint32_t used = (uint32_t)&_end - 0x20000000;
    uint32_t total = 20480;
    uint32_t free = total - used;

    printf("已用RAM: %lu 字节\n", used);
    printf("可用RAM: %lu 字节\n", free);
}
```

## 历史问题：32KB缓冲区方案

### 问题（2026-03-10发现）

原设计试图分配32KB缓冲区：
```c
❌ 旧代码（物理不可能）
#define CART_SERVICE_UPLOAD_BUFFER_SIZE (32 * 1024)
uint8_t upload_buffer[32768];  // 32KB

问题：
  32KB > 20KB总RAM
  差额：12,288字节（超出60%）
  结果：物理不可能！
```

### 解决方案：流式写入架构

```c
✅ 新代码（可行）
typedef struct {
    uint8_t sector_buffer[512];  // 仅512字节
    // ... 其他状态字段
} CartServiceRamJob;

数据流：
  Host → USB → FAT16 → 512B缓冲 → 卡带硬件
                         ↓
                    实时写入（1KB块）
```

**效果**：
- 内存需求：32KB → 620字节
- 减少：98%
- 状态：不可能 → 可行

**相关提交**：
- `0a62939`: fix: replace 32KB buffer with streaming writes
- `068bbb8`: docs: update RAM upload workflow
